03 ‚Äî Detailed Breakdown

Step-by-Step Timelines

Every task inside each module, as designed by the development team.

Module 1 ‚Äî AI Processing Layer Offline

Structural learning pipeline: from raw document ingestion to a structured, searchable WI Repository.

Step 1

Text Input Collection (GP / DDR)

Collect historical Game Plans and Daily Drilling Reports in PDF, Word, and Excel formats as the raw input to the pipeline.

Step 2

Text & Table Extraction

Parse documents to extract textual content and tabular data separately, handling scanned documents via OCR when needed.

Step 3

Text Summarization & Sentence Embeddings

Generate summaries and convert text into vector embeddings for semantic search and similarity matching.

Step 4

BERT / Fine-tuned LLM Processing

Apply domain-specific BERT or fine-tuned LLM models to understand drilling terminology and classify content. Uses OCR branch for scanned/handwritten documents.

Branch OCR if scanned / handwritten
Step 5

Contextual History & Document Reduction

Build contextual history from the processed documents and reduce redundancy to create a clean, deduplicated corpus.

Step 6

Records Value Extraction

Extract structured field values from records using field matching and pattern recognition to populate the data model.

Input Field Match techniques
Step 7

Records Validation

Validate extracted records using LLM-based event reporting. Documents passing as GP or DDR continue; others loop for vector DB retrieval.

Decision Reported Events (LLM-Based) ‚Äî if not GP/DDR ‚Üí Vector DBs for Retrieval
Step 8

Location Similarity (Semantic)

Match locations across documents using semantic similarity to group related operations regardless of naming conventions.

Step 9

GP & DDR Mapping

Align planned GP steps with actual DDR execution text to enable direct comparison and deviation analysis.

Step 10

Outcome Supply

Classify each mapped step's outcome as one of the four categories.

Completed Deviation Aborted Diverted
Step 11

Register Master Portal

Final output to the Master Portal ‚Äî a human-in-the-loop validation interface for engineers to accept or reject AI-generated alignments.

Via Field Match vs. DS Based comparison

Module 2 ‚Äî DWI Creation & Pre-Job Governance Interactive

Template-based workflow creation with 5-step logic definition and governance controls.

Step 1

Template Selection

Users select from pre-approved Execution Templates where the core logic structure is fixed, ensuring consistency and compliance.

Step 2

Pre-Assembly & Variable Entry

Populate the template with job-specific variable parameters ‚Äî depth targets, thresholds, severity levels, and equipment specifications.

Step 3

Define Pre-Conditions & Steps

Set entry conditions and define the execution steps. Optionally approve with comment for auto-flagging.

Gate Approve with comment (auto, or flagging)
Step 4

5-Step Logic Definition

Implement the core logic builder covering all five dimensions of instruction logic.

1 Operation Identification (Drill, RIH, POOH, etc.)
2 Step Behavior (Action, Monitor, Verify, Safety, Hold)
3 Constraints & Acceptance Criteria
4 Real-Time Evidence Linkage (Torque/RPM signals)
5 Dynamic Logic (Condition keys & event scopes)
Step 5

Validation & Consistency Check

Automated validation of the complete instruction set for logical consistency, missing references, and conflicting rules. Flagging for manual review when needed.

Step 6

Process Template Version

Version the validated template. Once signed, the DWI becomes a "Frozen Execution Package" ‚Äî locked and immutable for deployment.

Output Frozen Execution Package ‚Üí deployed to RT Engine

Module 3 ‚Äî Online Real-Time (RT) Engine Live

Live data interpretation and instant validation against the frozen plan.

Step 1

Connect to Live Data Points

Establish connections to rig sensor data streams via WITS, WITSML, and OPC protocols. Includes MQTT and KAFKA message brokers for reliable transport.

WITS WITSML OPC MQTT KAFKA
Step 2

Step Recognition / State Tracking

Automatically detect the current operation, phase, and sub-phase using signal patterns and historical context.

Step 3

Plan vs. Actual Deviation

Real-time monitoring of sequence, time, and parameters against the frozen execution plan. Alerts on deviations.

Step 4

Real-Time Execution Validation

Timestamp matching with surface data to validate every step as it happens. Uses KDE + Variance algorithms for statistical validation.

Engine KDE + Variance LI + Algo Checking
Step 5

KPI Board & Driller Interface

Push active instructions and KPI metrics to the driller's tablet on the rig floor. Plot high-risk events directly in the RT log.

Step 6

Archive to Database

Persist all execution data, validations, and events to the archive for post-job analytics and historical reference.

Module 4 ‚Äî Reporting & Post-Job Analytics Analytics

Automated reporting, deviation analysis, and feedback loop for continuous improvement.

Step 1

Collect Raw / Clean Data

Aggregate all execution data from the RT Engine ‚Äî sensor readings, validation results, deviation logs, and event timestamps.

Step 2

Deviation & Funnel Analytics

Perform statistical analysis on deviations. Create funnel views showing how planned operations compare to actual execution at each stage.

Step 3

Auto-Report Generation

Generate Daily Operation Reports instantly based on RT engine data. Includes operational KPIs, fatigue analysis, and template compliance metrics.

Operational KPIs Fatigue Analysis Template Compliance
Step 4

Mapping Module Integration

Cross-reference analytics with the Module 1 mapping layer to enrich historical context and improve future template accuracy.

Step 5

Feedback Loop Output

Implement the continuous improvement mechanism ‚Äî refine WI templates and update risk tagging based on post-job findings.

Step 6

Archive & Deliver to Module 1

Close the loop: archive all analytics results and feed enriched data back to Module 1's AI processing layer for continuous learning.

Loop Feeds back into Module 1 for re-training & improvement
